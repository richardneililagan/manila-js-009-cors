The problem with cross-domain requests

Server-to-server communication
is (generally) OK.

  • Not strictly cross-domain
  • Does not deal with XMLHttpRequest

Some client-to-server communication is OK

  • Cross-domain resources via HTML tags. <img src="http://some.domain">
  • Generally anything not using XMLHttpRequest

So what now?

Server Proxy

JSONP

Flash middleman

CORS
Cross-Origin Resource Sharing

W3C Standard mechanism for accessing cross-site resources previously controlled by the same-domain policy restriction.

CORS can be used to mitigate

  • Cross-domain XMLHttpRequests
  • Cross-domain webfont access, generally via CSS @font-face
  • WebGL textures
  • Images drawn to a canvas using drawImage
  • Simple
  • Preflighted
  • Credentialed

Simple requests

  • HTTP GET, POST, and HEAD requests
  • HTTP POST requests must use any of the following Content-Types:
    • application/x-www-form-urlencoded
    • multipart/form-data
    • text/plain
  • No custom headers

Notable request headers

  • Origin
  • Referer

CORS response headers

  • Access-Control-Allow-Origin

Preflighted requests

  • Most anything that's not simple
  • The client sends an HTTP OPTIONS preflight request before the actual request

Notable HTTP OPTIONS request headers

  • Origin
  • Access-Control-Request-Method
  • Access-Control-Request-Headers

CORS HTTP OPTIONS response headers

  • Access-Control-Allow-Origin
  • Access-Control-Allow-Methods
  • Access-Control-Allow-Headers
  • Access-Control-Max-Age

Credentialed requests

  • Non-simple requests that contain cookie data
  • NOTE : By default, cross-site requests do not contain cookie information

Additional CORS HTTP OPTIONS response headers

  • Access-Control-Allow-Credentials
